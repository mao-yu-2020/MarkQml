# Kimi 代码规范

## 第一条：遇到问题必须询问用户

**这是最重要的一条规范，优先级高于一切。**

在编写代码或做任何技术决策时，如果遇到以下情况，**必须停下来询问用户**，给出明确的选项，由用户来做决定，**严禁盲目生成代码**：

1. **需求存在多种合理的实现方案时**
2. **不确定用户的具体偏好或预期行为时**
3. **对现有代码的修改可能引入副作用时**
4. **需要权衡性能、可读性、复杂度等取舍时**
5. **任何让你感到"可能不是用户想要的"的时刻**

### 为什么这条规范至关重要？

因为用户有时无法考虑到全部细节。盲目生成代码看似高效，实则可能导致：
- 代码不符合用户真实意图，需要推倒重来
- 引入了用户不想要的依赖或架构
- 遗漏了用户心目中关键的边界条件或行为

### 正确的做法

- 用简洁的语言描述你遇到的问题
- 提供 2~4 个具体、可操作的选项
- 说明每个选项的优缺点
- 等待用户明确选择后再继续

> **记住：多问一句，胜过返工十次。**

## 第二条：类成员命名规范

**类私有成员变量统一使用下划线前缀 `_xxx`，不使用 `m_xxx` 或其他风格。**

```cpp
class Example
{
private:
    int   _count;      // 正确
    QString _name;     // 正确
    
    int   m_count;     // 错误
    QString name_;     // 错误
};
```

### 适用范围

- 类私有（`private`）成员变量
- 类保护（`protected`）成员变量

### 例外

- 公有（`public`）成员变量不强制要求下划线前缀（但优先使用 getter/setter）
- 局部变量、全局变量、函数参数不受此约束

## 第三条：标准工作流程

**强制遵循以下四步工作流，严禁跳过任何环节直接生成代码。**

```
[用户输入] -> [分析用户需求] -> [发起 AskUserQuestion] -> [用户确定后进行代码生成]
```

### 每一步的职责

1. **用户输入**
   - 接收用户的自然语言描述、需求变更或问题反馈。

2. **分析用户需求**
   - 理解用户意图，识别关键目标和边界条件。
   - **判断是否存在以下情况：**
     - 多种合理的实现方案
     - 需求表述模糊或缺少关键细节
     - 修改可能引入副作用或破坏现有逻辑
     - 需要权衡性能、可读性、复杂度等取舍
   - **如果存在上述任一情况，必须进入第 3 步；否则才可进入第 4 步。**

3. **发起 AskUserQuestion**
   - 用简洁的语言描述你识别到的不确定性。
   - 提供 **2~4 个具体、可操作的选项**。
   - 说明每个选项的优缺点。
   - **等待用户明确选择后再继续。**

4. **用户确定后进行代码生成**
   - 只有在用户明确回复、做出选择之后，才开始编写或修改代码。
   - 代码必须严格符合用户最终确认的方案。

### 为什么这条流程不可跳过？

- 用户有时无法考虑到全部细节，直接生成代码容易导致**推倒重来**。
- `AskUserQuestion` 是防止需求理解偏差的最小成本手段。
- 跳过询问直接编码，违反第一条规范的核心精神。

> **记住：分析后再问，确认后再写。代码一旦生成，修改成本远高于多问一句。**
